# 错误码处理方案

## 一、方案概述

### 1.1 后端职责

- 错误码定义为 `number` 类型 `code`
- 维护 2 份国际化 JSON（zh、en），格式示例如下
- 通过 Claude skill，在修改错误码相关逻辑时自动同步更新 zh、en 文件

**zh.json 示例：**

```json
{
  "100001": "参数错误",
  "100002": "请求超时，请稍后重试",
  "100010": "用户未登录",
  "100011": "登录已过期，请重新登录",
  "100020": "服务暂时不可用",
  "100021": "请求频率过高，请稍后再试",
  "100024": "模型请求错误",
  "100030": "资源不存在",
  "100031": "权限不足",
  "100040": "网络异常，请检查网络连接",
  "100050": "文件上传失败",
  "100051": "文件格式不支持"
}
```

**en.json 示例：**

```json
{
  "100001": "Invalid parameters",
  "100002": "Request timeout, please try again later",
  "100010": "User not logged in",
  "100011": "Session expired, please login again",
  "100020": "Service temporarily unavailable",
  "100021": "Too many requests, please try again later",
  "100024": "Model request error",
  "100030": "Resource not found",
  "100031": "Permission denied",
  "100040": "Network error, please check your connection",
  "100050": "File upload failed",
  "100051": "File format not supported"
}
```

### 1.2 前端职责

- 错误码 JSON 托管在 CDN
- 页面打开时异步拉取对应语言的文件
- 在 API 层实现拦截：若接口返回的 `code` 存在于错误码 JSON 中，则自动映射并提示；否则不拦截，由业务自行处理

### 1.3 为什么不放在前端

| 原因 | 说明 |
|------|------|
| 沟通成本 | 后端需单独告知前端错误码含义，增加协作成本 |
| 职责独立 | 错误码信息与前端的业务国际化完全独立，不应耦合 |
| 性能 | 错误码或未来新增国家后，若并入前端 i18n 会导致 bundle 或加载体积过大，影响首屏性能 |

---

## 二、方案优点

1. **职责清晰**：后端统一维护错误码及文案，前端只负责展示
2. **解耦**：错误码与前端业务 i18n 完全分离
3. **性能**：错误码文案不打包进前端，通过 CDN 按需拉取
4. **可扩展**：后续新增语言只需维护新的 JSON，前端改动小
5. **自动化**：Claude skill 辅助维护，减少人工遗漏

---

## 三、优化建议

### 3.1 加载时序

**问题**：首次进入页面时，若 API 先于错误码 JSON 返回失败，拦截器可能尚未拿到映射，无法展示友好文案。

**建议**：

- 在应用入口（如 layout、_app）尽早发起错误码 JSON 的预拉取
- 关键接口调用前可等待错误码加载完成
- 未加载完成时使用兜底文案，如 `"未知错误 (100024)"` 或 `"Unknown error (100024)"`

---

### 3.2 JSON 解析与 key 类型

**问题**：JSON 中数字 key `100024` 会被解析为字符串 `"100024"`，需保证查找时类型一致。

**建议**：

- 统一以字符串作为 key
- 拦截器中用 `String(code)` 查找，避免类型不匹配导致查不到

---

### 3.3 CDN 缓存与更新

**问题**：后端更新错误码文案后，CDN 可能仍返回旧内容。

**建议**：

- 使用带版本或时间戳的 URL，如 `/error-codes/zh.v2.json` 或 `?t=20250101`
- 或设置合理的 CDN 缓存 TTL（如 5–10 分钟）

---

### 3.4 拉取失败兜底

**问题**：网络异常或 CDN 不可用时，无法获取错误码映射。

**建议**：

- 拉取失败时重试 1–2 次
- 失败降级：仅展示 `code`，或使用通用提示「网络异常，请稍后重试」
- 可选：内置少量核心错误码的兜底文案，确保关键错误仍有可读提示

---

### 3.5 缓存策略

**问题**：每次打开页面都拉取，存在重复请求。

**建议**：

- 首次拉取成功后，在内存或 sessionStorage 中缓存
- 使用 ETag/Last-Modified 做协商缓存，避免重复传输
- 若文案有更新，通过 URL 版本或时间戳控制刷新

---

### 3.6 JSON 结构扩展（可选）

**当前格式**：`{ "100024": "模型请求错误" }`

**可选扩展**（按需演进）：

```json
{
  "100024": {
    "message": "模型请求错误",
    "action": "请稍后重试",
    "retryable": true
  }
}
```

适用于需要区分操作提示、是否可重试等更复杂场景。初期可先使用简单字符串格式。

---

### 3.7 未知错误码监控

**建议**：

- 拦截器遇到未在 JSON 中定义的 `code` 时，打日志或上报
- 便于发现遗漏维护或新增错误码，形成闭环

---

### 3.8 语言与 URL 关联

**问题**：前端需根据当前语言拉取对应 JSON。

**建议**：

- 复用现有 locale 配置（如 next-intl 的 `locale`）
- URL 示例：`/error-codes/${locale}.json` 或 `/error-codes/zh.json`、`/error-codes/en.json`

---

### 3.9 Claude Skill  scope

**建议**：明确 skill 的覆盖范围：

- 新增错误码时：自动在 zh、en 中补充
- 修改错误文案时：自动更新对应 key
- 删除错误码时：从 zh、en 中移除（或标记 deprecated）

---

### 3.10 错误信息安全性

**建议**：

- 错误文案中不暴露敏感信息（如内部路径、账号、配置等）
- 上线前由后端或安全侧做一次文案审核

---

## 四、总结

整体方案合理，职责划分清晰，易维护、易扩展。在补充加载时序、兜底策略、CDN 缓存与版本、失败重试和监控等优化后，可满足生产环境使用。
